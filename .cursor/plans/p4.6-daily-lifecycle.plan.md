# Phase 4.6: 日运行生命周期 — 分步执行计划

> **状态**: ✅ 已完成 (2026-03-22)
> **目标**: 模拟盘"开机即用、关机无忧" — 达到同花顺模拟盘级别的日常可用性

## 设计理念

用户的一天应该是：

1. **开盘前打开程序** → 自动检查数据 → 增量更新 → 恢复昨日状态 → 就绪
2. **盘中** → 行情自动拉取 → 订单自动撮合 → 策略自动运行 → 无需手动操作
3. **收盘后** → 自动结算 → 同步当日收盘数据 → 生成交易摘要
4. **关闭程序** → 无需担心，状态已持续保存在DB

## 核心决策


| 决策                                                    | 理由                     |
| ----------------------------------------------------- | ---------------------- |
| 状态持久化用 **已有的 sim_orders/sim_positions/sim_account 表** | Phase 3 已建表但未接入写逻辑     |
| 启动恢复从 DB 读取而非 Redis                                   | DB 是持久化存储，Redis 只做消息总线 |
| 数据自检在 **FastAPI lifespan** 中执行                        | 后端启动时自动完成，前端无需关心       |
| 收盘结算用 **scheduler 定时触发**                              | 不依赖外部 cron，保持自包含       |
| 交易日历从 **trade_cal 表** 读取                              | 已有数据，无需额外接口            |


## Step 1: OMS 状态持久化 (写入 + 恢复)

**目标**: 每次订单/持仓/资金变化时自动写入DB，后端重启后自动恢复

**后端改动**:

- `execution/persistence.py` (新文件):
  - `save_order(order)` → 写入 sim_orders 表 (INSERT or UPDATE)
  - `save_position(position)` → 写入 sim_positions 表
  - `save_account(account)` → 写入 sim_account 表
  - `load_all_state()` → 从DB读取全部状态，返回 (orders, positions, account)
  - 全部使用 async SQLAlchemy
- `execution/engine.py` (修改):
  - `submit_signal()` 成功后 → 调用 `save_order()`
  - `on_bar()` 成交后 → 调用 `save_order()` + `save_position()` + `save_account()`
  - `cancel_order()` 后 → 调用 `save_order()` + `save_account()`
  - 新增 `async restore_from_db()` → 启动时调用 `load_all_state()` 重建内存状态
- `execution/api.py` (修改):
  - `reset_account()` → 同时清空DB中的 sim_orders / sim_positions / sim_account

**关键约束**:

- 写DB操作不能阻塞主链路，用 `asyncio.create_task()` 异步写入
- 恢复时只恢复 SUBMITTED/PARTIAL_FILLED 状态的订单为活跃订单
- sim_account 表已有 1 行 (id=1)，用 UPSERT 语义

**验收**: 后端启动 → 下单 → 关闭后端 → 重启 → 订单/持仓/资金仍在

## Step 2: 启动自检 (数据 + 状态 + 日历)

**目标**: 后端启动时自动完成所有准备工作，无需手动操作

**后端改动**:

- `core/startup.py` (新文件):
  - `async startup_checks()` → FastAPI lifespan 中调用
  - 步骤:
    1. 检查 trade_cal 最新日期 vs 今天 → 决定是否需要更新日历
    2. 检查 stock_daily 最新日期 vs 上一个交易日 → 决定是否需要增量同步
    3. 如果数据过时 → 自动调用 sync_incremental 逻辑 (非阻塞，允许跳过)
    4. 调用 `trading_engine.restore_from_db()` 恢复OMS状态
    5. 日志输出自检结果摘要
  - `is_trade_date(date)` → 查 trade_cal 表判断是否交易日
  - `last_trade_date()` → 返回最近一个交易日
- `main.py` (修改):
  - 将 startup_checks() 集成到 FastAPI lifespan
  - 前端通过 `/health` 可看到自检状态

**数据自检规则**:

- stock_daily 滞后 > 1 个交易日 → 自动增量更新 (Tushare daily + daily_basic)
- stock_limit 滞后 > 1 个交易日 → 自动增量更新
- trade_cal 不含本月 → 自动更新
- 增量更新失败 (如Tushare接口异常) → 警告但不阻塞启动，用已有数据继续

**验收**: 后端启动 → 日志显示"数据自检: stock_daily 更新至 20260322, OMS 恢复 3 笔订单"

## Step 3: 盘中自动化 (调度器自启 + 撮合联动)

**目标**: 交易时段内行情自动拉取 → 订单自动撮合，无需任何手动操作

**后端改动**:

- `execution/feed/scheduler.py` (修改):
  - 启动时自动判断当前是否交易日+交易时段 → 自动开始/等待
  - watch_codes 从 **sim_positions + sim_orders 的持仓/挂单股票** 自动收集
  - 新增: 当有新订单提交时，自动将该股票加入 watch_codes
  - 盘中每分钟: 拉 Tushare rt_min → 发 Redis → `trading_engine.on_bar()` 撮合 (已实现)
  - 非交易日: 完全休眠不调用 Tushare
- `core/startup.py` (修改):
  - 自检完成后 → 自动启动 scheduler (无需手动 POST)
  - 自动恢复上次运行中的策略 (从DB或配置读取)

**交易日历集成**:

- scheduler 在启动时和每日 0:00 检查今天是否交易日
- 非交易日 (周末/节假日) → 不启动行情拉取，日志输出"今日非交易日"
- 这解决了"周末启动也尝试拉数据"的问题

**验收**: 交易日启动后端 → 9:30 自动开始拉数据 → 挂单自动被撮合 → 无需任何手动操作

## Step 4: 收盘结算 + 日终数据同步

**目标**: 15:00 自动结算当日交易，同步收盘数据，生成当日摘要

**后端改动**:

- `execution/feed/scheduler.py` (修改):
  - 15:00 后触发 `trading_engine.end_day()` → 结算当日盈亏
  - 15:30 后触发日终数据同步:
    - stock_daily + daily_basic (当日收盘数据)
    - stock_limit (当日涨跌停)
    - suspend_d (当日停复牌)
  - 结算结果写入 DB
- `execution/observability/daily_summary.py` (修改/新增):
  - 生成当日交易摘要: 成交笔数/总金额/盈亏/风控触发次数
  - 摘要写入 audit_log 或独立表
- 前端 (可选增强):
  - Dashboard 或 HistoryPage 显示"今日摘要"卡片

**日终同步规则**:

- 只同步当天数据，不全量重拉
- 如果已经同步过 (trade_date 已存在) → 跳过
- 同步失败 → 记录告警，不影响明天运行

**验收**: 15:00 后日志显示"结算完成: 成交5笔, 今日盈亏+2,340" → stock_daily 有当日数据

## 依赖关系

```
Step 1 (持久化) ── 独立，最先做
Step 2 (启动自检) ── 依赖 Step 1 (需要 restore_from_db)
Step 3 (盘中自动化) ── 依赖 Step 2 (需要 startup_checks)
Step 4 (收盘结算) ── 依赖 Step 3 (需要 scheduler 已运行)
```

严格串行，每步依赖上一步。

## 不做什么 (避免扩范围)

- 不改前端布局/组件 (只改后端生命周期)
- 不新增 DB 表 (复用已有的 sim_orders/sim_positions/sim_account/audit_log)
- 不做分钟数据自动同步 (太慢，单独手动跑)
- 不做通知推送 (微信/邮件，留给 Phase 5+)
- 不做多账户 (当前只有 1 个模拟账户)
